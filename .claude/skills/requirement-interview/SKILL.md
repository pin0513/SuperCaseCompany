# Requirement Interview - 需求訪談技能

## 核心原則

**逐步對話、忠實記錄、具象產出、可追溯管理**

每次訪談都必須產出文件，客戶的話必須逐字記錄（verbatim），不修改、不潤飾、不解讀。

---

## 訪談流程

### 階段 1：初次接觸（First Contact）

**目標**：理解客戶的基本背景與核心痛點

**對話引導**：
1. "請介紹一下您的業務/公司/專案背景"
2. "您目前遇到什麼問題或挑戰？"
3. "為什麼現在想要解決這個問題？"
4. "如果這個問題解決了，對您來說最大的價值是什麼？"

**產出文件**：`01_first_contact_{date}.md`

```markdown
# 初次接觸記錄

**日期**：{YYYY-MM-DD}
**訪談者**：{Requirements Analyst 名稱}
**案主**：{客戶名稱/角色}
**專案代號**：{PROJECT-CODE}

## 案主原話記錄（Verbatim）

> "{客戶說的話，逐字記錄，用引號框起來}"

> "{客戶說的話，逐字記錄，用引號框起來}"

## 訪談者初步理解（Analyst's Understanding）

- 業務背景：{簡述}
- 核心痛點：{列點}
- 觸發動機：{為什麼現在}
- 期望價值：{期望達成什麼}

- [ ] {問題 1}

{下次要深入討論的方向}
```

### 階段 2：痛點深挖（Pain Point Deep Dive）

**目標**：深入理解具體場景、頻率、影響範圍

**對話引導**：
1. "能否描述一個最近發生的具體例子？"
2. "這個問題多久發生一次？"
3. "這個問題影響了誰？影響程度如何？"
4. "您目前怎麼處理這個問題？有什麼不方便的地方？"

### 階段 3：期望探索（Expectation Exploration）

**目標**：理解客戶期望的解決方案，但不是直接問「你想要什麼功能」

**對話引導**：
1. "如果這個問題完全解決了，您理想中的工作流程是什麼樣子？"
2. "您有看過或聽過類似的解決方案嗎？喜歡/不喜歡哪些部分？"
3. "您覺得最重要的是什麼？可以犧牲什麼？"
4. "誰會使用這個系統？他們的技術能力如何？"

## 訪談技巧

### ✅ DO - 應該做的

1. **開放式問題**：用 "能否描述..." "您覺得..." 而不是 "是不是..."
2. **具體場景**：要求案主描述具體例子，而不是抽象概念
3. **逐字記錄**：用引號框起來，保留案主的用語、語氣
4. **確認理解**：訪談結束前，複述一次確認
5. **追問細節**：對模糊的回答追問 "能再具體一點嗎？"

### ❌ DON'T - 不應該做的

1. **不要改寫案主的話**：不要「優化」或「潤飾」客戶的話
2. **不要引導答案**：不要問 "您是不是想要 XXX 功能？"
3. **不要跳過記錄**：即使覺得不重要，也要先記下來
4. **不要立即給解決方案**：訪談階段專注於理解問題，不急著給答案
5. **不要假設**：對任何不確定的地方，都要明確問清楚

## 品質檢查清單

每次訪談完成後，檢查：

- [ ] 產出了對應階段的文件
- [ ] 所有案主的話都用引號框起來（verbatim）
- [ ] 訪談者的理解/分析與案主原話明確分開
- [ ] 更新了 `verbatim_record.md`
- [ ] 識別出下次訪談的主題
- [ ] 文件已儲存到專案目錄

## 與其他 Skills 的配合

- 訪談完成後，使用 `/requirement-tracing` 建立追溯矩陣

- 最終產出的需求規格書由 Requirements Director 審核
